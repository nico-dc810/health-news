# 大健康 AI 资讯雷达改版备忘

日期：2026-07-01

项目地址：https://github.com/nico-dc810/health-news

线上地址：https://nico-dc810.github.io/health-news/

## 一句话定位

大健康 AI 资讯雷达是一个面向大健康从业者的资讯筛选与行业情报入口，帮助用户快速看到政策、产业、产品、渠道、内容趋势中的重要信号，并为后续日报、选题、合规复核和 Agent 工作流预留接口。

## 改版背景

原项目功能较多，包含知识库、Agent、内容生成、合规审核、资讯抓取、工作台等完整业务闭环。作为首版上线产品，功能显得过重，用户进入后很难立刻理解核心价值。

本次改版的策略是先收敛为一个简单、清晰、可上线的资讯类网站。首页不再强调复杂工作台，而是先做一个类似资讯雷达的产品：

- 先让用户看到“今日重点信号”
- 先服务大健康从业者的信息筛选需求
- 先保留后续 Agent 和知识库接口位置
- 先用静态页面上线 GitHub Pages，降低部署复杂度

## 关键困境

### 1. 产品定位过宽

最初页面名称偏泛 AI，例如 “AI News Radar”，容易让人误以为这是一个通用 AI 新闻站，而不是服务大健康行业的资讯产品。

处理方式：

- 主品牌改为“大健康 AI 资讯雷达”
- 英文标识改为 “Health Intelligence Radar”
- 文案从“连接 IMA 知识库”改为“大健康从业者资讯筛选和行业情报”

### 2. 首版功能不能太重

原有系统具备很多后续能力，但首版如果全部暴露，会影响上线速度和用户理解。

处理方式：

- 首页只保留资讯雷达体验
- 登录页保留，用于之后权限和商业化
- Agent、日报、同步接口保留为占位，不做复杂交互
- 复杂后端暂不暴露给用户

### 3. GitHub Pages 只能托管静态站

GitHub Pages 不适合直接安全连接 IMA 或其他需要密钥的知识库接口。

处理方式：

- 首版页面做成纯静态站
- 前端请求 `/api/health/signals?limit=20`，接口不存在时自动使用本地 fallback 数据
- 后续可以通过后端服务、GitHub Actions 定时生成 JSON、或边缘函数提供真实数据

### 4. 部署链路不稳定

过程中遇到几个发布问题：

- 本地 `.git` 目录一开始是空壳，需要重新初始化
- GitHub CLI 安装成功，但登录接口多次超时
- GitHub Actions 的 Pages 工作流第一次失败，失败点在 `Configure Pages`
- 最终使用 `gh-pages` 分支发布静态文件，成功上线

最终方案：

- `main` 分支保存项目源码和文档
- `gh-pages` 分支保存 `static` 目录展开后的静态站
- GitHub Pages 设置为从 `gh-pages / root` 发布

### 5. 手机端显示问题

初版手机端分类按钮纵向堆叠，导致文章卡片被挤到很下面，用户误以为“手机端没有文章”。

处理方式：

- 分类栏改为横向滑动
- 顶部统计区改为响应式网格
- 文章卡片使用 `auto-fit + minmax` 自动从双列变单列
- 增加 1100、860、640、380 等响应式断点

## 当前前端数据方案

前端优先请求：

```text
GET /api/health/signals?limit=20
```

如果接口不可用，自动使用 `fallbackSignals` 演示数据。

单条信号建议字段：

```json
{
  "rank": 1,
  "category": "industry",
  "source": "行业资讯",
  "source_count": 3,
  "priority": "B",
  "published_at": "2026-07-01T08:00:00+08:00",
  "title": "标题",
  "summary": "摘要",
  "why": "为什么重要",
  "tags": ["政策", "产品", "内容趋势"],
  "url": "https://example.com/original"
}
```

## 后续接口预留

### 1. 行业资讯同步

用于把外部资讯、知识库条目或人工整理内容同步成统一的 Signal。

```text
GET /api/health/signals
POST /api/health/sync
```

后续可接入的数据来源：

- IMA 知识库
- RSS / 行业网站
- 政策公开网站
- 企业官网 / 竞品动态
- 自媒体与社媒热点
- 人工录入的行业观察

### 2. 日报生成

把当天信号整理成日报。

```text
POST /api/radar/daily-report
```

建议输出：

- 今日摘要
- 重点政策
- 产业趋势
- 产品机会
- 内容选题
- 合规风险
- 建议行动

### 3. Agent 处理

把单条信号交给后续 Agent 做加工。

```text
POST /api/agents/run
```

可支持的 Agent：

- 摘要生成
- 选题提炼
- 合规复核
- 来源引用
- 竞品分析
- 老板内参
- 小红书/公众号选题建议

### 4. 登录权限

保留手机号登录入口，后续可接真实权限。

```text
POST /api/auth/phone-login
```

后续可扩展：

- 企业账号
- 订阅状态
- 试用权限
- 管理员白名单
- 付款状态

## 推荐技术方案

### 首版静态方案

适合当前阶段。

架构：

```text
GitHub Pages
  └─ static/index.html
  └─ static/styles.css
  └─ static/app.js
```

优点：

- 上线快
- 成本低
- 维护简单
- 适合验证定位和页面体验

限制：

- 不能安全保存 IMA API Key
- 不能直接做后端计算
- 动态数据需要外部服务或静态 JSON

### 第二阶段：静态页面 + 定时 JSON

适合轻量自动化。

架构：

```text
GitHub Actions / 本地脚本 / 云函数
  -> 拉取资讯和知识库
  -> 生成 static/data/signals.json
  -> GitHub Pages 前端读取 JSON
```

前端读取：

```text
GET /data/signals.json
```

优点：

- 仍然可以托管在 GitHub Pages
- 不需要长期运行后端
- 适合日报类产品

限制：

- 数据不是实时
- 写入和权限控制弱

### 第三阶段：前端 + 后端 API

适合正式运营。

架构：

```text
GitHub Pages / Vercel / Cloudflare Pages
  -> 前端页面

Render / Railway / Cloudflare Workers / 自有服务器
  -> API 服务
  -> IMA 知识库
  -> 数据库
  -> 定时任务
```

建议后端能力：

- 数据源管理
- 知识库同步
- 信号去重
- 标签分类
- 摘要生成
- 风险识别
- 日报生成
- 用户权限

## 当前待注意事项

- GitHub CLI 已安装，但认证链路曾多次超时，不应依赖它作为唯一发布方式
- 目前 Pages 使用 `gh-pages` 分支发布，`main` 上的 Actions 工作流失败不影响当前线上访问
- 旧项目文件仍在本地工作区中，但没有全部提交到 GitHub
- IMA 仍可作为后端数据来源，但不应作为前台卖点直接暴露给用户
- 静态站里的登录目前是前端演示逻辑，不具备真实安全权限控制

## 当前结论

本次改版的核心不是“做完一个复杂平台”，而是先上线一个清楚、有行业定位、能被目标用户理解的资讯入口。

第一版应该让用户马上知道：

- 这是给大健康从业者看的
- 它帮我筛重要资讯
- 它告诉我为什么重要
- 后面可以继续生成日报、选题和合规建议

复杂能力可以慢慢接入，但首页必须先清楚、轻、快。
